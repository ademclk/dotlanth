# ADR-26031: Opcode Determinism Classification Model

## Status
Proposed (target decision for v26.3.0-alpha)

## Context
- Deterministic enforcement cannot stay implicit inside scattered host code.
- Dotlanth must avoid aggressive opcode growth; metadata should explain policy, not new opcodes.
- Strict mode needs a stable way to decide whether an opcode is allowed, constrained, or rejected.

## Decision
- Every opcode relevant to strict execution is assigned determinism metadata with one category:
  - `pure`
  - `controlled_side_effect`
  - `non_deterministic`
- Category semantics:
  - `pure`: deterministic given the same inputs and runtime state; allowed in strict mode.
  - `controlled_side_effect`: may touch externalized state, but behavior is allowed only through declared capabilities and deterministic policy checks; audited when used.
  - `non_deterministic`: cannot satisfy the v26.3 strict contract; rejected in strict mode.
- Metadata also carries any required capability and policy notes needed by enforcement/tooling.
- Unclassified opcodes fail closed in strict mode.

## Alternatives Considered
1) Binary allowed/blocked flag - rejected: too weak for tooling and future policy evolution.
2) Large taxonomy with many subtypes - rejected: premature complexity and policy sprawl.
3) Capability-only model - rejected: capabilities answer permission, not determinism.

## Consequences
### Positive
- Runtime, tooling, and docs share one vocabulary.
- Classification can evolve without changing the user-facing language surface.
- The release stays metadata-first instead of opcode-first.

### Negative / Tradeoffs
- Classification requires disciplined coverage tracking.
- Some edge cases will still need policy notes beyond the top-level category.
