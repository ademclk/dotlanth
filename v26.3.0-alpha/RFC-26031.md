# RFC-26031: Opcode Determinism Classification and Strict Constraints

## Status
Draft (target for v26.3.0-alpha)

## Summary
Defines the opcode determinism categories used by strict mode and the constraint model for each category.

## Goals
- Give runtime, artifacts, and tooling one shared classification vocabulary.
- Keep policy metadata-first so Dotlanth avoids opcode surface growth.
- Make strict-mode gaps fail closed.

## Categories (normative)

### `pure`
- Deterministic for the same inputs and relevant runtime state.
- Allowed in strict mode.

### `controlled_side_effect`
- May affect externalized or persisted state.
- Allowed in strict mode only when:
  - declared capability is present
  - operation passes deterministic policy checks
  - audit/reporting is emitted

### `non_deterministic`
- Not allowed in strict mode.
- MUST be denied before host execution.

## Metadata Requirements
- Each in-scope opcode entry MUST expose:
  - category
  - capability requirement, if any
  - optional policy notes for enforcement/tooling
- Missing metadata MUST be treated as unsupported in strict mode.

## Coverage Rule
- CI or tests MUST make classification gaps visible.
- New in-scope opcode behavior MUST not silently inherit permissive strict-mode behavior.
