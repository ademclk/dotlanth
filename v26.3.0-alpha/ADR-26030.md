# ADR-26030: Strict Deterministic Execution Contract

## Status
Proposed (target decision for v26.3.0-alpha)

## Context
- v26.2.0-alpha added artifact bundles and replay tooling, but replay does not yet prove consistency.
- Dotlanth needs an execution contract that lets operators distinguish replayable deterministic runs from best-effort re-execution.
- ERA 4 steering requires determinism to strengthen the execution fabric, not to add framework behavior.

## Decision
- Add an opt-in strict deterministic execution mode.
- In strict mode, runtime behavior fails closed on any prohibited non-deterministic operation.
- Strict mode is recorded in run metadata, artifacts, and replay validation outputs.
- Strict mode composes with capabilities:
  - capability declaration remains required for controlled side effects
  - determinism policy adds an additional gate and can still deny execution

## Alternatives Considered
1) Best-effort warnings only - rejected: weakens replay credibility.
2) Always-on strict determinism - rejected for v26.3: too disruptive before coverage is classified.
3) Per-feature ad-hoc deterministic toggles - rejected: policy becomes fragmented and framework-like.

## Consequences
### Positive
- Replay claims become testable instead of aspirational.
- Operators get a clear signal for whether a run was eligible for validation.
- Future policy modes can build on a stable contract.

### Negative / Tradeoffs
- Some existing examples will fail under strict mode until classification and hooks are complete.
- Fail-closed behavior may feel stricter than current replay users expect.

## Rollout Notes
- v26.3 ships strict mode as opt-in.
- Non-strict execution continues to work, but artifacts should clearly state that replay validation is limited or unsupported.
