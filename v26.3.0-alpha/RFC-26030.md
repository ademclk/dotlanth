# RFC-26030: Deterministic Execution Mode Spec

## Status
Draft (target for v26.3.0-alpha)

## Summary
Defines the runtime contract for deterministic execution modes in v26.3.0-alpha, with emphasis on the new strict mode.

Strict mode exists to make replay validation meaningful. It is a runtime policy, not a framework feature.

## Goals
- Provide an explicit execution mode that fails closed on unsupported non-deterministic behavior.
- Make mode selection visible in runtime records, artifacts, and CLI output.
- Create a stable base for replay validation.

## Non-Goals
- No always-on strict determinism requirement in v26.3.
- No cryptographic attestation.
- No new language surface beyond minimal mode selection UX.

## Modes

### `default`
- Existing execution behavior remains available.
- Artifacts should indicate replay validation may be unsupported or limited.

### `strict`
- Runtime MUST apply deterministic policy before host execution.
- Unclassified or prohibited operations MUST fail closed.
- Determinism mode MUST be persisted in run metadata and artifact metadata.

## CLI Surface
- Proposed invocation:
  - `dot run --deterministic=strict <entry.dot>`
- Tooling MAY later add shorthand aliases, but the explicit form is the normative v26.3 contract.

## Failure Semantics
- Strict mode denials MUST occur before forbidden side effects execute.
- Error output MUST identify the failure as a deterministic policy violation.
- Failed strict runs MUST still emit required artifact sections with explicit error markers.

## Artifact Requirements
- Bundle metadata MUST include:
  - `determinism_mode`
  - validation eligibility status
  - violation summary
