# RFC-26033: Replay Proof and Validation Contract

## Status
Draft (target for v26.3.0-alpha)

## Summary
Defines the replay proof payload stored in artifacts and the comparison behavior for `dot validate-replay <run>`.

## Goals
- Make replay validation machine-checkable without requiring byte-for-byte bundle equality.
- Keep the proof surface narrow, stable, and explicit.

## Validator Outcomes
- `valid` - replay proof matches the original proof surface.
- `invalid` - replay completed but proof comparison mismatched.
- `unsupported` - original run lacks required proof material or was not eligible for validation.

## Proof Surface (normative)
- determinism mode
- validator eligibility
- opcode classification summary
- deterministic violation summary
- capability usage / denial summary
- supported state diff summary
- deterministic trace-shape facts needed for comparison
- budget summary fields chosen for v26.3 validation

## Artifact Requirements
- Artifact bundle MUST store proof material in a stable machine-readable form.
- Failed strict runs MAY be ineligible for successful validation, but MUST still record eligibility and failure markers explicitly.

## CLI Contract
- `dot validate-replay <run>` MUST:
  - locate the original run
  - perform replay using the supported replay path
  - compare proof material
  - print one stable status line and actionable details on mismatch/unsupported cases
