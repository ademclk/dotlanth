# ADR-26032: Replay Validator Proof Surface and Artifact Metadata

## Status
Proposed (target decision for v26.3.0-alpha)

## Context
- v26.2 replay can rerun inputs but cannot assert whether the rerun matches the original execution contract.
- Byte-for-byte bundle comparison is too broad and brittle for alpha.
- The validator needs a narrow, stable, machine-readable proof surface that works with artifact bundles.

## Decision
- `dot validate-replay <run>` compares a documented non-cryptographic replay proof surface.
- Proof material is embedded in the artifact bundle and includes stable summaries/fingerprints for:
  - determinism mode
  - opcode classification summary
  - deterministic violation summary
  - capability usage / denial summary
  - supported state diff summary
  - deterministic trace shape facts needed for validation
- Validator outputs one of `valid`, `invalid`, or `unsupported`.
- `unsupported` is used when the original run lacks required proof material or was not eligible for deterministic validation.

## Alternatives Considered
1) Compare full bundle bytes - rejected: brittle, over-coupled to incidental layout.
2) Compare logs only - rejected: too weak and not artifact-first.
3) Delay validation until cryptographic proofs exist - rejected: blocks core operator value.

## Consequences
### Positive
- Replay validation becomes reliable enough for alpha without overpromising security guarantees.
- Artifact bundles remain the primary validator input.
- Proof material can evolve version-by-version.

### Negative / Tradeoffs
- Non-cryptographic proofs do not provide tamper resistance.
- Narrow proof surfaces can miss unsupported differences unless documented clearly.
