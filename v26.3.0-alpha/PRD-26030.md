# PRD-26030: Dotlanth v26.3.0-alpha - Deterministic Mode + Replay Validation

## Summary
Dotlanth v26.3.0-alpha delivers PHASE 2 of the ERA 4 roadmap: an opt-in strict deterministic execution mode backed by opcode classification, boundary enforcement, artifact-level determinism metadata, and replay validation.

This release does not try to become a framework. It makes Dotlanth more execution-fabric-like by turning replay from a convenience into a verifiable contract.

## Problem
v26.2.0-alpha established the Artifact-First Runtime: every run emits an artifact bundle with trace, state diff, and capability reporting. That baseline is necessary, but it is not yet sufficient for trustworthy replay.

Current gaps:
- Replay can rerun inputs, but it cannot prove the replay is execution-consistent.
- Time, randomness, and network access can make two runs diverge with no first-class deterministic policy.
- Capabilities are explicit, but determinism policy is not yet explicit at opcode level.
- Tooling can inspect a run, but it cannot answer "did replay match the original deterministic contract?"

For an AI-native execution fabric, reproducibility must be a runtime contract with artifacts and validation, not a best-effort hope.

## Goals
- **G1 (DET I):** Add a strict deterministic execution flag that fails closed on prohibited operations.
- **G2 (OP Classification I):** Classify opcodes into `pure`, `controlled_side_effect`, and `non_deterministic`, with explicit constraints.
- **G3 (SEC II):** Enforce determinism rules at the opcode/host boundary and emit audit events for every violation.
- **G4 (VM III):** Add deterministic hooks for time, random, and network gating; start determinism budgeting.
- **G5 (TL III):** Add `dot validate-replay <run>` and make deterministic mode visible in CLI UX.
- **G6 (AF II):** Extend artifact bundles with determinism metadata and non-cryptographic replay proofs.

## Non-Goals
- NG1: No new connectors or external service packs.
- NG2: No large new opcode families; classify and constrain what already exists first.
- NG3: No SDK generation or client libraries.
- NG4: No distributed consensus, remote attestation, or cryptographic proof chains.
- NG5: No performance-led work that compromises determinism or artifact clarity.
- NG6: No framework features, web runtime abstractions, routing DSLs, or middleware systems.

## Target Users
- Maintainers who need deterministic runtime contracts that survive replay.
- Tooling authors who need machine-checkable replay validation.
- Security reviewers who need to see where deterministic policy was enforced or violated.
- Agent operators who need to know whether a run is safe to treat as reproducible.

## User Stories (high-level)
- As a maintainer, I want strict mode to reject non-deterministic execution so replay claims stay credible.
- As a tooling author, I want opcode classification metadata so validator behavior is explicit and extensible without opcode growth.
- As an operator, I want `dot validate-replay <run>` to tell me whether the replay matches the original proof surface.
- As a security reviewer, I want deterministic violations logged as audit events with source context.
- As an agent, I want determinism metadata in the artifact bundle so I can choose between safe automation and manual review.

## Requirements

### Functional
- **R1: Strict deterministic flag**
  - CLI/runtime support a strict deterministic execution flag.
  - Accepted surface should be explicit and stable, for example `--deterministic=strict`.
  - The chosen mode is recorded in run metadata and artifact bundle metadata.
- **R2: Opcode classification**
  - Every opcode reachable in strict mode is classified as one of:
    - `pure`
    - `controlled_side_effect`
    - `non_deterministic`
  - Classification lives in opcode metadata, not scattered policy code.
  - Unclassified opcode usage in strict mode fails closed.
- **R3: Strict mode rules**
  - `pure` opcodes are allowed in strict mode.
  - `controlled_side_effect` opcodes are allowed only when:
    - required capability is declared
    - operation stays inside deterministic policy constraints
    - audit/reporting is emitted
  - `non_deterministic` opcodes are rejected in strict mode before side effects execute.
- **R4: Deterministic host hooks**
  - Time access is gated through a deterministic policy hook.
  - Randomness access is gated through a deterministic policy hook.
  - Network access is gated through a deterministic policy hook.
  - Violations are surfaced consistently through runtime errors + audit events.
- **R5: Boundary enforcement and audit events**
  - Determinism checks happen before host execution.
  - Each violation emits an audit event with run id, opcode, category, capability context, and trace linkage where possible.
- **R6: Replay validator**
  - `dot validate-replay <run>` loads the original run, performs replay, and compares a defined proof surface.
  - Validator reports one of:
    - valid
    - invalid (proof mismatch)
    - unsupported (original run not eligible for validation)
  - Validator exits non-zero for invalid and tooling errors.
- **R7: Proof surface**
  - Replay proof compares stable execution facts, including at minimum:
    - deterministic mode
    - opcode classification summary
    - trace shape relevant to deterministic enforcement
    - state diff summary within supported scope
    - capability usage / denial summary
  - Proof format is non-cryptographic but stable and machine-readable.
- **R8: Artifact bundle extension**
  - Bundle adds determinism metadata section(s), including:
    - execution mode
    - validator eligibility
    - violation counts
    - proof summary / fingerprints
  - Failed strict runs still emit required metadata with explicit failure markers.
- **R9: Budgeting start**
  - Runtime begins tracking a small deterministic budget surface for future enforcement, such as:
    - gated hook counts
    - denied non-deterministic attempts
    - controlled side-effect counts
  - v26.3 stores and reports budget facts; hard budget policy can remain future work.

### Non-functional
- **Determinism credibility**
  - Strict mode must fail closed rather than silently downgrade.
  - Replay validation must compare a documented canonical surface.
- **Security**
  - Determinism enforcement composes with capability enforcement; neither bypasses the other.
  - Audit events must be present even for failed executions.
- **Stability**
  - Classification names and proof schema must be versioned/documented.
  - CLI output should be concise and scriptable.
- **DX/UX**
  - Error messages should explain whether failure came from missing capability, deterministic violation, or validator mismatch.

## Hard Constraints (must not violate)
- Dotlanth remains an execution fabric; framework-like changes are rejected.
- Artifacts remain first-class outputs.
- Determinism is implemented by policy, metadata, and validation, not by growing product surface area.
- Avoid aggressive opcode growth and avoid premature optimization.

## Done When
- Build/quality gates:
  - `cargo fmt --check`
  - `cargo clippy --workspace --all-targets --all-features -- -D warnings`
  - `cargo test --workspace`
- Strict mode contract works:
  - safe deterministic examples pass under strict mode
  - non-deterministic examples fail before forbidden side effects execute
- Validator contract works:
  - `dot validate-replay <run>` reports valid for a deterministic-safe run
  - mismatch path is test-covered and produces actionable output
- Artifact contract works:
  - bundle includes determinism metadata and replay proof fields
  - failed strict runs still yield required artifact sections

## Demo Flow (v26.3 alpha)
1) Run a deterministic-safe example:
   - `cargo run -p dot -- run --deterministic=strict examples/deterministic/basic.dot`
2) Inspect the run:
   - `cargo run -p dot -- inspect <run_id>`
3) Validate replay:
   - `cargo run -p dot -- validate-replay <run_id>`
4) Run a non-deterministic example in strict mode and confirm rejection.
5) Export artifacts if needed and inspect determinism metadata.

## Metrics / Signals
- Percentage of in-scope opcodes classified.
- Percentage of strict-mode failure paths that emit audit events + artifact metadata.
- Replay validator success/mismatch coverage in tests.
- Operator time to answer "is this run reproducible enough to trust?"

## Risks
- **RISK-1:** Over-classifying too early creates churn.
  - Mitigation: classify only reachable/in-scope opcodes and fail closed for the rest.
- **RISK-2:** Replay validator compares too much and becomes flaky.
  - Mitigation: define a narrow proof surface in ADR-26032.
- **RISK-3:** Determinism rules duplicate capability rules in confusing ways.
  - Mitigation: keep capability and determinism separate but composed at the same boundary.
- **RISK-4:** Team reaches for new opcodes/connectors to "fix" deterministic gaps.
  - Mitigation: explicit non-goal and ADR guardrails.
