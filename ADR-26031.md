# ADR-26031: Opcode determinism classification semantics

## Status
Accepted

## Context
Dotlanth strict mode needs one determinism vocabulary that runtime enforcement, tooling, and future validators can share. Before this milestone, strict mode relied on a syscall allowlist, which made policy decisions harder to explain and easier to drift from registry metadata.

## Decision
In-scope opcodes are classified with one of these categories:

- `pure`
- `controlled_side_effect`
- `non_deterministic`

The categories mean:

- `pure`: no host-side effect and no capability requirement; strict mode allows these opcodes
- `controlled_side_effect`: host interaction is explicit, capability-gated, and policy-visible; strict mode allows these opcodes when the required capability is granted
- `non_deterministic`: the opcode can observe or create environment-dependent behavior that is outside the current strict execution contract; strict mode rejects these opcodes

Strict mode is fail-closed for missing coverage. If an in-scope opcode lacks classification metadata, execution is rejected before the opcode runs.

## Current in-scope mapping

- `halt` => `pure`
- `load_const` => `pure`
- `mov` => `pure`
- `log.emit` => `controlled_side_effect`, requires `log`
- `net.http.serve` => `non_deterministic`, requires `net.http.listen`

## Consequences

- Runtime enforcement and tooling can describe opcode policy with the same terms.
- New in-scope syscall ids must be classified explicitly before strict mode will accept them.
- Controlled side effects stay available for narrow, capability-gated use cases without widening strict mode to nondeterministic operations.
