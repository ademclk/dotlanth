# RFC-26031: Opcode classification registry and strict-mode enforcement

## Summary
This milestone introduces a shared opcode classification model for the current VM/runtime surface and uses it to make strict mode fail closed for missing classification.

## Registry shape

- `dot_ops` owns the in-scope syscall registry and shared `DeterminismClass`
- `dot_vm` owns opcode-level metadata exposure for VM instructions and composes syscall metadata into the opcode surface

Each in-scope opcode exposes:

- classification category
- required capability context, when applicable

## Enforcement rules

- Pre-execution strict validation walks the built VM program and rejects:
  - any opcode classified as `non_deterministic`
  - any syscall usage missing classification metadata
- Host dispatch keeps a strict-mode runtime backstop so manual or future call paths still fail closed for unclassified syscall ids

## Coverage strategy

- registry-driven tests assert that every in-scope opcode exposes determinism metadata
- strict-mode tests cover:
  - pure opcode acceptance
  - controlled side-effect acceptance
  - non-deterministic syscall rejection
  - unclassified syscall rejection

## Current classifications

- VM core:
  - `halt`
  - `load_const`
  - `mov`
  - all classified as `pure`
- Syscalls:
  - `log.emit` => `controlled_side_effect`
  - `net.http.serve` => `non_deterministic`
