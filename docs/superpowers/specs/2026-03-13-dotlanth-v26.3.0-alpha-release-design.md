# Dotlanth v26.3.0-alpha Release Design

**Goal:** Deliver deterministic execution mode and replay validation for `v26.3.0-alpha` through reviewable milestone PRs that stay aligned with the provided release packet.

## Scope

This release implements the provided `v26.3.0-alpha/` packet in dependency order:

1. `M1-DET`: strict deterministic mode plumbing and fail-closed semantics
2. `M2-OPC`: opcode determinism classification and coverage enforcement
3. `M3-VM`: deterministic hooks and budget counters
4. `M3-SEC`: composed determinism/capability boundary enforcement and audit events
5. `M4-AF`: artifact determinism metadata and replay proof surface
6. `M5-TL`: replay validator and deterministic CLI UX
7. `M6-DOC`: end-to-end docs, demo, and release gate

The user approved splitting the combined `M3-VMSEC` milestone into separate PRs where that improves reviewability.

## Constraints

- Follow the release packet as the source of truth.
- Keep diffs scoped to determinism mode, classification, replay validation, and release docs.
- Do not expand runtime product surface beyond what the packet requires.
- Do not claim universal determinism; claim strict-mode validation over a supported proof surface.
- Preserve the existing crate seams unless a small shared type is needed to carry determinism metadata.

## Architecture

### Existing seams to preserve

- `crates/dot`: CLI entrypoints, scripted UX, inspect/validator output
- `crates/dot_rt`: runtime context shared between parsed documents and execution
- `crates/dot_vm`: opcode metadata, syscall dispatch, VM events
- `crates/dot_ops`: host boundary where time/random/network side effects enter
- `crates/dot_sec`: policy vocabulary and enforcement composition
- `crates/dot_db`: persisted run metadata and bundle lookup
- `crates/dot_artifacts`: bundle manifest schema and writer

### Shared release vocabulary

The release should converge on a small shared set of types rather than ad hoc strings:

- `DeterminismMode` with at least `default` and `strict`
- opcode-level determinism classification metadata
- deterministic denial/audit vocabulary shared by VM, host, artifacts, and CLI
- replay proof summary types stored in artifacts and consumed by the validator

For `M1-DET`, mode visibility is satisfied by bundle manifest metadata plus stable `dot inspect` output. Persisting mode into DotDB run rows is not required in this milestone.

### Enforcement boundary

Determinism policy and capability policy meet at the VM-to-host boundary, but they remain separate concerns:

- capability policy answers whether a declared permission exists
- determinism policy answers whether the operation is allowed in the selected mode

This separation keeps later policy changes from tangling security and replay semantics.

## Delivery Topology

### PR order

1. `release/v26.3.0-alpha-m1-det`
2. `release/v26.3.0-alpha-m2-opc`
3. `release/v26.3.0-alpha-m3-vm`
4. `release/v26.3.0-alpha-m3-sec`
5. `release/v26.3.0-alpha-m4-af`
6. `release/v26.3.0-alpha-m5-tl`
7. `release/v26.3.0-alpha-m6-doc`

### Branching rule

Each PR branches from the latest merged release branch, not from an older local stack. This keeps every review focused on the current milestone contract.

### Worktree rule

Use isolated worktrees when implementing milestone PRs. For this release, use a global worktree root under `~/.config/superpowers/worktrees/dotlanth/` to avoid polluting the repository root while multiple milestone branches are active.

## Verification Strategy

Each PR proves its own contract plus the direct contracts it depends on:

- `M1-DET`: mode selection, metadata persistence, fail-closed behavior
- `M2-OPC`: classification completeness and strict-mode missing-coverage denial
- `M3-VM`: policy-aware hooks and budget counters
- `M3-SEC`: denial-before-dispatch and audit-event emission
- `M4-AF`: artifact determinism metadata and replay-proof payloads
- `M5-TL`: validator outcomes, exit codes, and deterministic inspect/run UX
- `M6-DOC`: real `run -> inspect -> validate-replay` flow

Verification should stay narrow during development and expand to milestone-level commands before creating each PR.

## First Slice

Implementation starts with `M1-DET`.

`M1-DET` is responsible for introducing the shared `DeterminismMode` vocabulary, plumbing it from CLI into runtime execution, surfacing it through persisted run or artifact metadata, and denying unsupported strict-mode behavior before side effects execute. It should not attempt full opcode coverage or replay proof generation; those belong to later milestones.

The concrete fail-closed rule for `M1-DET` is: in `strict` mode, all host syscalls are denied before dispatch unless and until later milestones classify and explicitly allow them. This keeps `M1` narrow while still delivering a real strict-mode contract.
